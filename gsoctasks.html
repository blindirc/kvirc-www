Google Summer of Code 2008 - KVIrc Ideas List

This is a list of potential projects for students participating in Google Summer of Code 2008. It has been submitted to Google for approval along with the KVIrc project itself. Once (and if) it's approved, you will be able to submit student applications to Google.

We have 5 potential tasks, listed in order of increasing difficulty. All of them unconditionally require:

  • the capability of using the subversion tool
  • capability of building kvirc from sources on at least one supported platform
  • availability for interaction with the other developers via mailing list and IRC.
  • decent reading and writing in English.
Each entry has a list of additional skills required to complete the task.

Task list

  • Code cleanups and documentation
  • User help subsystem cleanup
  • Cross-platform build system
  • User-friendly script/addon packaging system
  • Native Qt4 port

Code cleanups and documentation

Abstract

The kvirc source code is becoming quite large. Junior developers are discouraged by its complexity and the lack of documentation about it. All the functionality must be guessed from source code analysis. Recently we have started documenting the source code with the Doxygen tool. We have also estabilished a self-documenting coding standard that should be followed in the source tree. The goal of this task is to

  • enforce the usage of coding standards in the whole source tree
  • add the missing Doxygen documentation for the core classes

Details

Coding standard enforcement

The student will be required to run through the whole source tree and:

  • Spot the places where improper variable and class names are used and fix them in order to enforce the usage of hungarian notation to the degree estabilished by our coding standard.
  • Evaluate the usage of global variables and possibly replace them with singletons.
  • Move the standalone C functions into C++ namespaces
  • Enforce other "good practice" programming patterns

Doxygen documentation

We ask the student to add doxygen based documentation for all the classes contained in the kvilib library and for the most important classes in the kvirc executable. For each documented class we require at least an overall description and the documentation of the most important member functions (purpose, parameters and return value).

Required additional skills

  • Basic knowledge of C++

Difficulty

Easy

User help subsystem cleanup

Abstract

KVIrc has an internal help system that provides user documentation about the program interface and scripting commands. The documentation text is extracted from the source code by the means of a perl script. The script produces html output that is distributed with the executable and displayed to the user by the means of a simple builtin html viewer. The perl script has been modified and adjusted several times in the past and has become unmantainable. The documentation produced also has several broken links and extraction defects. The goal of this task is to rewrite the documentation extraction tool (either in perl or maybe another portable language) in order to make it mantainable and produce an improved set of user help documents. The task may also involve modifying the help viewer to match the changes made on the extractor tool.

Details

The actual document extraction tool is a batch converter that processes the cpp sources given by the commandline. It parses the files looking for special kind of comments and produces a set of html files that are later presented to the user by the means of the internal help viewer and distributed via web. The internal help viewer is actually implemented as a kvirc module. We ask the student to:
  • Define a new polished user documentation structure (indexes, subsections, topic groups etc...)
  • Rewrite the document extraction tool in order to match the structure defined above. The source code must be properly organized and commented in order to make in matainable in the future.
  • Review the documentation source for "hardcoded" broken links
  • Eventually adjust the internal help viewer to match the changes made in the document extraction tool

Required additional skills

  • Knowledge of perl or another language that can be used to extract documentation from sources.
  • Basic knowledge of HTML
  • Possibly a decent knowledge of C++

Difficulty

Medium

Cross platform build system

Abstract

Building KVIrc on all the supported platforms is a complex task since every platform has a separate set of configurations and tools for this purpose. Most developers support only a single platform and are not able to verify their changes on all the other build systems. The goal of this task is to explore the usage of the CMake build system along with its CPack extension in order to create a single consistent set of build configuration files for all the supported platforms.

Details

We're actually using
  • Automake/Autoconf-based build system on linux
  • Visual Studio Express project files on windows
  • A modified automake/autoconf setup on macosx.

We want to have a single unified build system on all the platforms. We have attempted to use qmake, the Qt build tool, but we aren't satisfied by the final results. The other candidate is CMake which seems to provide also a distribution subsystem called CSetup. The set of new build files should be created in a separate subdirectory of the source tree (say build/cmake) and the existing build systems shouldn't be removed until the new one is considered to be stable.

Required additional skills

  • Decent knowledge of C++
  • Experience in cross-platform application building (linux, windows and macosx).
  • CMake experience is a huge plus

Difficulty

Medium

User-friendly script/addon packaging system

KVIrc has a builtin scripting language that can be used to create various kinds of addons. The addons can include sets of new scripting commands or even graphical elements like specialized toolbars or windows. The distribution and user installation of such addons is non-trivial ad requires the user to extract archives to special paths, move files "by hand" and possibly edit configuration files with an editor. The goal of this task is to create a builtin addon packaging and installation system similar to the one that has been created for theme distribution.

Details

The kvirc svn tree contains the KVS sources for several addons. Some of these addons contain a non standardized "self-installation" procedure and other (in fact most of them) must be installed totally by hand. We already have the code for packing and unpacking generic kvirc data inside standard distributable binary packages. This packaging stystem has already been used to implement the distribution of kvirc themes. The themes can be both packaged for distribution and installed by using an user-friendly wizard-based graphical interface. We ask the student to implement a similar approach for the scripting addon subsystem by:
  • writing a graphical UI module that allows both packaging (export) and installation (import) of addons
  • defining a scripting standard that must be followed by the scripters in order to fully take advantage of the new packaging method.

Required additional skills

  • Good knowledge of C++
  • Knowledge of the KVIrc scripting subsystem.

Difficulty

Might be challenging

Native Qt4 port

Abstract

The current kvirc source code is strongly based on the Qt3 library from TrollTech. The newest Qt4 releases are incompatible with Qt3 and existing applications require a huge porting work. There is a very fragile and preliminary Qt4 support in kvirc that relies on the Qt3Support library provided by TrollTech. The Qt3Support library is meant as a temporary fix and will be dropped by TrollTech some time in the future. The goal of this task is to remove the dependency on Qt3Support and make kvirc a pure Qt4 application.

Details

We already brought the source tree to a point where it compiles with Qt4 and Qt3Support enabled. We also have an experimental branch in which we have attempted a native Qt4 port. The branch has important working changes but unfortunately has also non-working ones. We ask the student to:
  • Provide a native Qt4 implementation for the classes in the toolkit abstraction layer, eventually by backporting them from the experimental qmake branch to trunk.
  • Substitute the usage of all the Qt3Support classes with Qt4 native ones, eventually providing a wrapper implementation in the toolkit abstraction layer.
  • Fix the run-time issues caused by the Qt4 internal incompatbilities. This is meant particularly for the painting code that shows defects in font metric management and slowdowns due to multiple buffering layers.
  • Get kvirc to compile without using the Qt3Support library on all the supported platforms

Required additional skills

  • Strong knowledge of C++
  • Strong knowledge of Qt4 class hierarchy
  • Experience in porting applications from Qt3 to Qt4

Difficulty

Hard
